2. Context Transfer ‚Äî Switching Models or Rebuilding

If you ever need to switch AI providers, change models, rebuild the agent, or move to a new machine ‚Äî here's exactly how to transfer full context.

Why This Works

Forge's entire "brain" lives in files, not in conversation history. This means:

  • Any AI model can be briefed from these files in one session
  • No critical knowledge lives only in chat ‚Äî everything is written down
  • The workspace IS the agent. Transfer the workspace = transfer the agent.

3. Transfer Procedures

Scenario A: Switching API Key (Same Provider)

1Zero downtime ‚Äî context stays intact 
# Just swap the key
openclaw config set env.ANTHROPIC_API_KEY "sk-ant-NEW-KEY"
openclaw gateway restart

No context transfer needed. The workspace and all memory files are untouched.

Scenario B: Switching to Subscription Auth

1Same as above ‚Äî just change auth method 
# Switch to setup-token
openclaw models auth paste-token --provider anthropic
openclaw gateway restart

No context transfer needed. Same model, different billing.

Scenario C: Switching AI Provider (e.g., Anthropic ‚Üí OpenAI)

1Change the model in config 
openclaw config set env.OPENAI_API_KEY "sk-YOUR-KEY"
openclaw config set agents.defaults.model.primary "openai/gpt-4o"
openclaw gateway restart
2New model reads workspace files automatically

OpenClaw injects workspace files (AGENTS.md, SOUL.md, USER.md, etc.) into every session. The new model reads them on first message and picks up full context.

3Send a test message

Message the bot on Telegram: "What's my name, what's your name, and what are we building?" If it answers correctly, transfer is complete.

Key Insight
Because all context lives in files (not conversation history), switching providers is seamless. The new model reads the same files and immediately has full context. No manual briefing needed.

Scenario D: Moving to a New Machine (VPS Migration)

1On the OLD machine ‚Äî export workspace 
# Commit latest changes
cd ~/.openclaw/workspace
git add -A && git commit -m "Pre-migration backup"

# Create archive
tar -czf ~/forge-workspace-backup.tar.gz -C ~/.openclaw workspace/
2On the NEW machine ‚Äî install OpenClaw 
curl -fsSL https://openclaw.ai/install.sh | bash
openclaw onboard
3Restore workspace 
# Copy archive to new machine (scp, USB, whatever)
tar -xzf forge-workspace-backup.tar.gz -C ~/.openclaw/
4Start gateway and test 
openclaw gateway start
# Test on Telegram: "What's my name and what are we working on?"

Scenario E: Total Rebuild (Nuclear ‚Äî Agent From Scratch)

1Fresh install + restore files

Follow Scenario D steps. If the workspace backup is available, that's all you need. The new agent reads the files and becomes Forge.

2If NO backup exists (worst case)

The SOPs on your Google Drive are your secondary backup. Send them to the new agent one by one. Key files to recreate first:

  1. SOUL.md (personality)
  2. IDENTITY.md (name, role)
  3. USER.md (your info)
  4. MEMORY.md (rebuild from what you remember + daily logs if available)
  5. Company Foundation + all SOPs from Google Drive
This Is Why Google Drive Matters
Your Google Drive is the secondary backup of everything. If the Mac dies, the VPS crashes, and git is lost ‚Äî the PDFs on your Drive contain all the knowledge. You can rebuild from them.
