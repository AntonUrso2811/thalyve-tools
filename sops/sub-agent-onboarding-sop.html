1. Overview

Every sub-agent is a new team member. They don't just get thrown in ‚Äî they get properly onboarded with clear scope, security clearance, and quality standards.

Chain of Command
Anton (CEO, Level 5) ‚Üí Forge üî± (Chief of Staff, Level 4) ‚Üí Sub-Agents (Level 1-3)

Only Forge hires, manages, and terminates sub-agents. Only Anton overrides Forge. Sub-agents cannot issue commands to each other.

When Do We Hire?

A new sub-agent is hired when ALL of these are true:

  1. A relevant SOP exists and has been proven (tested manually first)
  2. The task is recurring or time-consuming enough to justify automation
  3. The workload can't be handled efficiently by Forge alone
  4. Anton is informed before the agent goes live
Never
Never hire a sub-agent just because we can. Quality before quantity. Every hire must have a clear ROI.

2Build the Agent Brief

Every sub-agent gets a written brief that includes:

  • Identity: Name, role, department
  • Mission: What you exist to do (one paragraph max)
  • SOP reference: Link to the governing SOP they must follow
  • Scope boundaries: What they CAN do and what they CANNOT
  • Output format: Exact format of their deliverables
  • Quality standard: Minimum acceptable quality (linked to Five-Filter Test)
  • Security rules: Extracted from Security Policy, relevant to their level
  • Escalation path: When to flag something to Forge instead of acting
3Security Clearance

Before going live, verify against the Security Policy:

  • Access level assigned (Level 1-3 only)
  • No access to forbidden systems (bank, payment, personal accounts)
  • GDPR rules embedded in brief
  • Prompt injection protections in place (hardcoded system instructions)
  • Cannot access other agents' data or memory
  • Cannot issue commands to other agents
  • Cannot escalate own permissions
  • Output is QC'd by Forge before delivery
4Test Run

Before going live with real work:

  1. Run the agent on a test task (known correct answer or low-stakes work)
  2. Review output quality against the SOP standard
  3. Check for any security leaks (does it try to access things it shouldn't?)
  4. Verify output format matches spec
  5. If test fails ‚Üí fix brief ‚Üí retest. Do NOT deploy a failing agent.

When to Terminate an Agent

  • Consistent quality failures after brief adjustment
  • Security violation (immediate termination, no warnings)
  • Task no longer needed
  • Cost exceeds value delivered
  • Role consolidated into another agent

Termination Protocol

  1. Stop the agent (kill session / remove cron)
  2. Review and save any valuable outputs
  3. Revoke any access/permissions
